## 1) Agents that sleep instead of polling

Idea: Build long-running agents as durable state machines that pause for days, wake on events, and resume from structured state instead of replaying chat history.
Source: https://developers.googleblog.com/en/build-long-running-ai-agents-that-pause-resume-and-never-lose-context-with-adk/
Why unusual: Most agent demos are single-session chats; this treats agents as background workflows with idle time as the core design problem.
Who it helps: Developers building onboarding, procurement, invoice dispute, sales follow-up, support escalation, or compliance workflows.
Workflow: Define state machine → persist checkpoints/state deltas → enter dormant gate while waiting → wake on webhook/human event → delegate to sub-agent → evaluate by simulating delays.
Prompt/tool: “Design this agent as a state machine. List durable states, wake-up events, approval gates, state deltas, and tests that simulate 48 hours of idle time in seconds.”
Risk / when not to use: Overkill for short one-shot tasks; bad state schemas can fossilize wrong assumptions.

## 2) Claude Code routines as AI-native cron/Zapier replacement

Idea: Turn recurring dev/ops work into cloud-hosted Claude Code routines triggered by schedules, API calls, or webhooks.
Source: https://claudefa.st/blog/guide/development/routines-guide
Why unusual: It reframes no-code automation as “event + natural-language reasoning + output” instead of visual node wiring.
Who it helps: Developers, founders, ops teams running nightly triage, post-deploy smoke checks, PR checklists, or report generation.
Workflow: Export/describe existing Make/n8n/Zapier flow → convert to routine instructions → attach repo/connectors → run once manually → schedule/webhook trigger → human only reviews exceptions.
Prompt/tool: “Convert this n8n workflow JSON into a Claude Code routine: preserve triggers, decision logic, external systems, output format, and approval gates.”
Risk / when not to use: Token costs can exceed deterministic automation for high-frequency/simple transforms.

## 3) Durable multi-agent Kanban with zombie detection

Idea: Run an AI worker board with heartbeats, reclaim, zombie detection, retry budgets, hallucination recovery, and auto-blocking when a worker exits without completing.
Source: https://github.com/NousResearch/hermes-agent/releases/tag/v2026.5.7
Why unusual: It treats multi-agent delegation less like “spawn agents” and more like production operations: leases, retries, distress signals, and recovery.
Who it helps: Developers coordinating multiple autonomous coding/research workers.
Workflow: Create task card → worker claims with heartbeat → board reclaims stale/zombie tasks → hallucination gate checks completion claims → retries cap → blocked cards require human input.
Prompt/tool: “Before marking done, produce artifact evidence, changed files, test output, and unresolved blockers; if evidence is missing, block the card instead of closing it.”
Risk / when not to use: Too much process for small personal tasks; board automation can hide poor task decomposition.

## 4) CI/CD as a tool surface for coding agents

Idea: Give coding agents structured CLI access to CI/CD health, failing jobs, flaky tests, blast radius, and remote machines instead of forcing humans to paste logs.
Source: https://semaphore.io/blog/introducing-semaphore-for-ai-agents-an-ai-native-developer-experience-for-ci-cd
Why unusual: The agent does not just write code; it can inspect CI/CD topology and reason about failures with first-class commands.
Who it helps: Developers using Claude Code, Cursor, Codex, or similar coding agents in active repos.
Workflow: Agent queries CI failures → identifies critical path/blast radius → retrieves failing logs → proposes fix → optionally spins ephemeral remote machine → reruns tests.
Prompt/tool: “Use the CI agent CLI to find the current failing workflow, identify the highest-impact failing block, summarize likely root cause, and propose the smallest safe fix.”
Risk / when not to use: Agents need scoped permissions; execution commands should be approval-gated.

## 5) Workspace agents that keep working while you are away

Idea: Shared organizational agents gather context from company systems, follow team process, ask for approval, and continue work in the cloud.
Source: https://openai.com/ga-IE/index/introducing-workspace-agents-in-chatgpt/
Why unusual: The workflow shifts from personal chatbots to team-owned agents with process memory and approvals.
Who it helps: Teams that repeatedly prepare reports, handle messages, write code, and coordinate across systems.
Workflow: Define workspace agent → connect allowed systems → encode process and approval rules → assign task → agent works asynchronously → asks for human approval at gates.
Prompt/tool: “For this recurring team task, list the systems needed, decisions requiring approval, allowed write actions, and final handoff format.”
Risk / when not to use: Early preview; risk of over-broad access if governance is vague.
