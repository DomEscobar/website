## Raw ideas

### 1. Voice notes as the fastest agent interface

- Source: Rost Glukhov — “I Built a Voice-Controlled AI Assistant on My Phone — No APIs Required” — https://medium.com/practical-llm-systems/i-built-a-voice-controlled-ai-assistant-on-my-phone-no-apis-required-0fc0b3d99b13
- Why unusual: The workflow uses normal phone voice messages as the input layer for a self-hosted agent. Local faster-whisper transcribes, the existing agent pipeline does the work, and Edge TTS replies with spoken audio — no paid APIs.
- Who it helps: Developers with self-hosted agents; normal people who want to capture ideas or trigger checks while away from the keyboard.
- Workflow: phone voice note → local transcription → route to agent → tool execution / research / triage → spoken response.
- Prompt/tool: “Check production health for the last hour and report only critical errors” as a 10–30 second voice message.
- Risk / when not to use: Bad for stack traces, complex multi-step specs, noisy audio, or actions that need careful review.
- Score: 7/7
- Could become post: yes

### 2. Phone mission control for coding agents

- Source: Marcelo Costa — “From Idea to Execution: Building an AI-Powered IDE Companion App with Antigravity and Gemini 3.1” — https://mesmacosta.medium.com/from-idea-to-execution-building-an-ai-powered-ide-companion-app-with-antigravity-and-gemini-3-1-60ae10bfd913
- Why unusual: A phone becomes a secure remote control and narrated status panel for a coding agent running in an IDE. The useful layer is not code generation, but observability while the agent runs.
- Who it helps: Developers running long agent tasks, test suites, or IDE-based agents while away from the desk.
- Workflow: IDE session → WebSocket tunnel → phone PWA streams state → lightweight AI summarizes current progress → user sends commands back.
- Prompt/tool: “Summarize current agent state in one sentence: writing, testing, stuck, or waiting for input.”
- Risk / when not to use: Remote-control security is high-risk; signing, replay protection, and strict permissions matter.
- Score: 6/7
- Could become post: yes

### 3. Orchestrator-only life assistant

- Source: Hector Flores — “I Repurposed a Coding Agent as a Life Assistant. Then My Twins Came 10 Weeks Early.” — https://htek.dev/articles/coding-agent-as-life-assistant-nicu
- Why unusual: The top-level assistant does not do the work. It only routes tasks to background sub-agents with their own context windows, so a stuck analytics job does not block a health reminder.
- Who it helps: Developers building serious personal agents; normal people indirectly via more reliable assistants.
- Workflow: Telegram/cron request → orchestrator classifies → launches background sub-agent → sub-agent completes independently → reports back.
- Prompt/tool: “Route this request to exactly one domain agent. Do not solve it in the main chat unless it is trivial.”
- Risk / when not to use: Requires clear routing rules and permissions; otherwise it becomes an invisible mess of background work.
- Score: 7/7
- Could become post: yes

### 4. Client projects with continuous domain agents

- Source: Hector Flores — “Two Client Sites in 3 Days: Agentive Context Engineering in Action” — https://htek.dev/articles/two-client-sites-three-days-agentive-context-engineering
- Why unusual: Instead of one coding agent session, each client/project gets a domain agent with isolated context, daily standups, cron work, preview URLs, and feedback loops.
- Who it helps: Freelancers, agencies, product builders, and developers juggling multiple similar projects.
- Workflow: discovery call → project-specific agent → hourly cron tasks → PR preview → client feedback → agent incorporates updates.
- Prompt/tool: “Create a domain agent for this project. Store brand rules, pending tasks, client preferences, and daily standup format.”
- Risk / when not to use: Client work needs strong review; agent speed can amplify wrong assumptions.
- Score: 6/7
- Could become post: yes

### 5. Google agents-cli: tools for coding agents, not another agent

- Source: google/agents-cli — https://github.com/google/agents-cli/
- Why unusual: The CLI gives existing coding agents skills for scaffolding, evaluating, deploying, and observing Google ADK agents. It is tooling for agents to build agents.
- Who it helps: Developers building deployable agents on Google Cloud or ADK.
- Workflow: coding agent + agents-cli skills → scaffold → run/eval → deploy → publish/observe.
- Prompt/tool: “Use agents-cli to scaffold an ADK agent, add evals, and prepare Cloud Run deployment. Stop before publishing.”
- Risk / when not to use: Google Cloud-specific; not useful if you are not building ADK/Gemini Enterprise agents.
- Score: 4/7
- Could become post: maybe, as a developer-only note

## Best candidate angle

Title: **Voice Notes Are the Best Interface for Small Agent Jobs**

Short answer: Don’t try to run your whole workday by voice. Use voice for tiny, operational jobs: status checks, idea capture, triage, and “what should I do next?” moments.

Flow:

```txt
voice note
→ local transcription
→ existing agent pipeline
→ tool call / check / draft
→ short spoken reply
```

Visual component idea:
- `FlowGraphic` for the voice-agent loop
- `CompareGrid` for “typing in chat” vs “voice command on phone”
- `DoDont` for good/bad voice tasks
- `KeyPoints` for voice prompt rules: one objective, short, explicit, constrained output

## Candidate prompt

```txt
Voice command rules for my assistant:
1. Treat this as one task only.
2. If the request is risky, summarize and ask before acting.
3. Reply with max 3 bullets.
4. If you need code, logs, or long text, ask me to switch to text.

Task: [spoken task]
```
