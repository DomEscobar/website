## Ideas

Idea: Partial harnesses beat over-scripted agent workflows
Source: arXiv — “Harnesses for Inference-Time Alignment over Execution Trajectories” https://arxiv.org/abs/2605.21516
Why unusual: The useful move is not “more process around agents”; the paper reports that fully structured workflows can hurt, while specifying only the initial steps and letting the agent execute the rest can improve pass rate.
Who it helps: Developers using Claude Code/Codex-style terminal agents; small teams writing agent runbooks; non-devs who over-template AI tasks.
Workflow: For a messy task, write only: 1) goal, 2) first evidence-gathering steps, 3) stop/check criteria, 4) allowed tools. Do not prescribe every edit. After the first checkpoint, let the agent propose the next plan from evidence.
Prompt/tool: “Start with these three checks only: [checks]. After that, summarize evidence, name the main uncertainty, and propose the smallest next action. Do not continue past the checkpoint without updating the plan.”
Risk / when not to use: Bad for regulated or destructive work where every step must be constrained; also risky if the agent lacks good verification tools.

Idea: Turn failed agent traces into reusable “skill text” instead of just retrying
Source: arXiv — “Trace2Skill: Verifier-Guided Skill Evolution for Long-Context EDA Agents” https://arxiv.org/abs/2605.21810
Why unusual: It treats natural-language instructions as an evolvable policy mined from success/failure traces, not as static prompting or model fine-tuning.
Who it helps: Developers with repeatable tests, CI, hidden/verifier-like failures, flaky migrations, or long-context repos.
Workflow: Run 3-5 attempts on a hard task; collect failure logs, test output, touched files, and successful recovery moves; distill them into a local `SKILL.md`/`CLAUDE.md` rule; rerun with that skill as part of the harness.
Prompt/tool: “Read these failed traces and produce: recurring failure modes, verifier-relevant files, recovery heuristics, and 5 permanent rules for future attempts. Keep rules operational and testable.”
Risk / when not to use: Can encode superstition if the verifier is weak or the traces are too few; sanitize any hidden tests/secrets before feeding logs back to agents.

Idea: Replace dashboards with tiny MCP snapshots agents can pull into work
Source: Hacker News Show HN — “I threw away my analytics dashboard and replaced it with 42 MCP tools” https://news.ycombinator.com/item?id=48233125
Why unusual: The creator optimized analytics for agent consumption: small token snapshots, `llms.txt`, read-only auth, and MCP/API access, rather than another human dashboard.
Who it helps: Indie hackers, marketers, product engineers, and normal site owners who want decisions inside their working chat instead of a separate analytics ritual.
Workflow: Create a read-only analytics MCP with tiny commands: `snapshot`, `top_pages`, `conversion_drop`, `compare_period`, `source_breakdown`. Add a project rule telling the coding/marketing agent when to call it before changing copy, SEO, or onboarding.
Prompt/tool: “Before changing this page, call analytics snapshot + top_pages + compare_period. Use the numbers to name one likely bottleneck, then propose one small experiment.”
Risk / when not to use: Easy to overfit to low-volume data; keep permissions read-only and avoid letting agents act on analytics without human approval for external changes.

Idea: Spec-driven context resets for coding agents
Source: Hacker News Show HN — “Spec-Driven Development Workflow for Claude Code” https://news.ycombinator.com/item?id=48231575 and repo https://github.com/sermakarevich/sddw
Why unusual: The key practice is deliberately clearing context between spec layers and implementation subtasks, using files as persistent memory instead of one giant chat.
Who it helps: Developers shipping medium-size features with coding agents; normal people can adapt it for documents, trip plans, legal/admin checklists, or renovations.
Workflow: Generate requirements → code/context analysis → design → subtasks as files. Clear chat between stages. Each new session reads only the relevant spec and writes back decisions/lessons.
Prompt/tool: “Create `requirements.md`, `analysis.md`, `design.md`, and `tasks.md`. After each file, stop. In the next context, read only the current file and the project rules, then implement one task.”
Risk / when not to use: Slower for tiny fixes; bad specs can fossilize misunderstandings, so each layer needs a quick human review.

Idea: Visual judge loops for taste-constrained design, not just code
Source: arXiv — “TO-Agents: A Multi-Agent AI Pipeline for Preference-Guided Topology Optimization” https://arxiv.org/abs/2605.21622
Why unusual: The pipeline translates fuzzy preference (“tree-like”, manufacturable, aesthetically aligned) into iterative solver parameters, then uses rendered multi-view critiques and an independent judge to revise.
Who it helps: Designers, makers, architects, product teams, and normal people doing home/interior/3D-print decisions where taste and constraints collide.
Workflow: For any visual/physical option, generate 3 candidates, render/photograph from multiple views, ask one agent to critique against preference and constraints, another to judge, then revise the generator parameters rather than only asking for a new image.
Prompt/tool: “Judge these options against: [taste], [hard constraints], [manufacturing/use constraints]. Return: score, violated constraints, which generation parameter to change, and one reason not to change it.”
Risk / when not to use: Judges can reward superficial style and miss physics, cost, accessibility, or safety; needs hard constraints and human review before fabrication/purchase.

## Pattern watch

- Strongest content angle: agent workflow design is moving from “prompt harder” to evidence contracts: partial harnesses, context reset, trace-mined skills, read-only MCP data, and verifier feedback.
- Best field-note candidate: Trace2Skill + partial harnesses combined into “stop over-orchestrating agents; evolve their local rules from evidence.”
- Normal-people bridge: dashboards and design tools become callable context inside the assistant, but only safely when read-only, tiny, and checkpointed.
