## Raw ideas

### 1. Figma-to-frontend agent pipeline with deterministic visual QA

- Source: WyeWorks — “The Workflow Is the Product: Building Specialized Pipelines for Coding Agents” — https://www.wyeworks.com/blog/2026/05/13/custom-agentic-workflows-for-coding-agents/
- Why unusual: Instead of asking one coding agent to “build this section,” they designed a specialized pipeline around the task. Separate agents handle Figma asset extraction, Astro/Tailwind implementation, framework validation, CSS compatibility, and computed-style checks against the source design.
- Who it helps: Frontend developers, agencies, designers working with devs, and anyone tired of “looks close enough” AI UI output.
- Workflow: Figma/design input → asset extractor → component analyst → implementer → Astro validator → CSS compatibility validator → browser computed-style comparison → review.
- Prompt/tool: “Do not judge visually by vibe. Extract expected spacing/type/size values, inspect browser-computed CSS, and report exact mismatches.”
- Risk / when not to use: Overkill for small UI tweaks; fragile if design files are messy or tokens are not explicit.
- Score: 7/7
- Could become post: yes

### 2. The lint/test/review gauntlet for AI coding agents

- Source: Steve Kinney — “My Ridiculous AI-Assisted Development Workflow” — https://stevekinney.com/writing/ai-assisted-development-workflow
- Why unusual: The “agent” is not the secret. The secret is a full operating system around it: Linear planning, dependency graph, worktrees, strict linters, tests, PR review bots, and a feedback loop where failures become rules.
- Who it helps: Developers using Claude Code/Codex/Cursor who want safer parallel work.
- Workflow: expert agents draft plan → junior agent critiques → dependency graph → worktrees → agents implement → lint/test gates → PR review bots → failure notes update rules.
- Prompt/tool: “Before implementation, produce dependency graph and file ownership. After review failure, write the exact guardrail that would have prevented it.”
- Risk / when not to use: Expensive and heavy; over-engineered for small tasks.
- Score: 6/7
- Could become post: yes

### 3. CI/CD as an agent interface, not a dashboard

- Source: Semaphore — “Introducing Semaphore for AI Agents” — https://semaphore.io/introducing-semaphore-for-ai-agents-an-ai-native-developer-experience-for-ci-cd
- Why unusual: CI/CD context is becoming structured input for coding agents. Instead of tab-switching into dashboards, agents can query pipeline state, debug failures, and collect build context through a CLI/interface.
- Who it helps: Developers debugging CI failures from inside Claude Code/Cursor/Codex.
- Workflow: coding agent asks Semaphore CLI → fetch pipeline/log context → summarize failure → suggest fix → run/check again.
- Prompt/tool: “Use CI context first. Do not guess the failure cause until you inspect the latest pipeline logs and changed files.”
- Risk / when not to use: Needs careful permissions; agents should not deploy or rerun expensive jobs without constraints.
- Score: 5/7
- Could become post: yes

### 4. Agency-in-a-box: heartbeat-driven agent teams

- Source: chenhg5/agencycli — https://github.com/chenhg5/agencycli
- Why unusual: Agents are defined as Markdown + YAML roles/projects, wake up on schedules, message each other via inbox, and coordinate through a lightweight CLI instead of a Python framework.
- Who it helps: Developers who want autonomous agent teams without writing orchestration code.
- Workflow: define org chart → apply project blueprint → scheduler wakes agents → agents drain task queues → wakeup routines proactively find work → human handles approvals.
- Prompt/tool: “Create a PM, dev, QA, and writer agent for this project. Each wakes on schedule, owns one queue, and escalates risky actions.”
- Risk / when not to use: Easy to create noisy busywork; strong wakeup routines and approval inboxes matter.
- Score: 6/7
- Could become post: yes

### 5. Agent boundary discovery for indie developers

- Source: Antigravity Lab — “Three Months with Antigravity Agents in My Indie Dev Workflow” — https://antigravitylab.net/en/articles/agents/antigravity-agents-decision-boundary-indie-developer
- Why unusual: The practical insight is that agents reveal which parts of your work are formalized. They work where rules are explicit; they stop where your judgment is tacit.
- Who it helps: Solo developers and creators deciding what to delegate.
- Workflow: list repeatable tasks → write explicit rules → delegate review/changelog/dependency analysis → keep taste/product decisions human → document every boundary where the agent stalls.
- Prompt/tool: “Tell me whether this task has explicit right/wrong criteria. If not, ask what judgment rule is missing before proceeding.”
- Risk / when not to use: Agents can produce technically correct but strategically wrong user-facing decisions.
- Score: 6/7
- Could become post: yes

## Best candidate angle

Title: **Stop Asking AI to “Make the UI Look Right”**

Short answer: AI frontend work gets reliable when “looks right” becomes measurable. Split the job into asset extraction, implementation, validation, and computed-style comparison.

Flow:

```txt
design source
→ extract assets + tokens
→ implement component
→ build/framework validation
→ browser computed-style checks
→ human review
```

Visual component idea:
- `FlowGraphic` for the UI pipeline
- `CompareGrid` for “vibe QA” vs “computed-style QA”
- `DoDont` for when the pipeline is worth it
- `KeyPoints` for exact checks: spacing, typography, dimensions, alignment, unsupported CSS

## Candidate prompt

```txt
Implement this UI section, but do not rely on visual vibe alone.

Return:
1. extracted design values: spacing, type, dimensions, colors, radius
2. implementation plan with component boundaries
3. browser-computed CSS checks after implementation
4. exact mismatches vs the source design
5. what still needs human taste review
```
