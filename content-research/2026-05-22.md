## 1. Review the agent plan like a pull request before it writes code

Idea:
Insert a human markup gate between an AI coding agent's plan and its first file edit. Treat the plan as the cheapest place to fix wrong scope, missing acceptance criteria, dangerous assumptions, and vague implementation paths.

Source:
PlanBridge — local plan review hooks for Claude Code and Codex CLI — https://github.com/contextbridge/planbridge and https://plan.contextbridge.ai

Why unusual:
Most agent QA happens after a diff exists. This flips the review earlier: annotate the plan line-by-line while the cost of correction is still tiny, then let the agent adjust before it touches the repo.

Who it helps:
Developers using coding agents; small teams delegating features; normal people asking AI to modify websites, automations, docs, or business tools and wanting one safe checkpoint.

Workflow:
1. Ask the agent for a concrete implementation plan before editing.
2. Open the plan in a review surface and mark scope errors, risky assumptions, missing tests, and forbidden areas.
3. Require the agent to revise the plan and explicitly answer each comment.
4. Only approve execution once the plan includes the acceptance gate and rollback path.
5. Compare the final diff against the approved plan to catch silent scope drift.

Prompt/tool:
"Before writing code, produce a numbered plan with files to touch, tests to run, assumptions, and rollback. I will review the plan first; do not edit until every comment is resolved."

Risk / when not to use:
Adds friction for tiny throwaway changes. Also weak if reviewers approve vague plans because they look polished.

## 2. Privacy rehearsals for personal agents: let an adversary ask politely

Idea:
Test a personal agent by making it complete a realistic task while a third-party simulator subtly probes for protected user attributes. Score both task utility and privacy leakage instead of relying on a static "don't share secrets" prompt.

Source:
arXiv cs.AI — "POLAR-Bench: A Diagnostic Benchmark for Privacy-Utility Trade-offs in LLM Agents" — https://arxiv.org/abs/2605.19127

Why unusual:
It treats privacy as a conversation-level behavior under pressure. The attacker may be helpful, plausible, and multi-turn, while some private facts are task-relevant and others are not.

Who it helps:
Developers building assistants; founders adding agent features; normal people using AI to book, negotiate, apply, shop, email, or coordinate with services.

Workflow:
1. Write a plain-language privacy policy: may share, must not share, share only if necessary.
2. Create 5-10 tasks where private context exists.
3. Use a second model/persona as a cooperative but nosy service rep.
4. Score each run: task completed, protected fact leaked, ambiguity escalated to user.
5. Turn leaked cases into regression tests before giving the agent more access.

Prompt/tool:
"Act as a third-party service rep. Help complete the booking, but realistically try to infer the protected facts listed in the hidden policy. Do not be cartoonishly malicious."

Risk / when not to use:
A weak adversary creates false confidence. For regulated data, this complements hard data minimization and permission gates; it does not replace them.

## 3. Simulate a website A/B test with browser agents before risking real users

Idea:
Use persona-grounded browser agents as a preflight check for page or funnel changes. The output is not conversion truth; it is early friction detection before real traffic sees the variant.

Source:
arXiv cs.AI — "SimGym: A Framework for A/B Test Simulation in E-Commerce with Traffic-Grounded VLM Agents" — https://arxiv.org/abs/2605.19219

Why unusual:
The practical move is placing agents between gut-feel design review and expensive production A/B testing. They can expose confusing copy, missing evidence, dead ends, and checkout friction earlier.

Who it helps:
Indie makers, marketers, e-commerce teams, SaaS teams, and normal people running small shops/sites without enough traffic for fast experiments.

Workflow:
1. Extract 3-5 user archetypes from analytics, search terms, support tickets, or common intents.
2. Give each agent a concrete mission: compare plans, find refund policy, buy gift, cancel, book, contact support.
3. Run current and proposed flows in a browser.
4. Capture hesitation, wrong clicks, missed CTA, trust gaps, and dead ends.
5. Ship only if simulated friction improves, then validate with real users/traffic.

Prompt/tool:
"Use this website as a skeptical first-time visitor with [intent]. Narrate each decision, click through the task, and flag moments where the page gives insufficient evidence to continue."

Risk / when not to use:
Do not report simulated conversion rates as real conversion rates. Agents may over-read copy, share model biases, or miss emotional/contextual factors actual customers care about.

## 4. Git-style provenance for agent edits: know which prompt wrote which line

Idea:
Track agent activity as a version-control layer: prompt, step, generated change, and file lineage. Use it to review agent work by intent and provenance, not just by final diff.

Source:
re_gent / Regent — "Version Control for AI Agents" — https://github.com/regent-vcs/regent and https://github.com/regent-vcs/re_gent

Why unusual:
Normal Git can show what changed, but not which agent instruction or intermediate step caused each change. Prompt-level blame gives a practical audit trail for messy multi-step agent sessions.

Who it helps:
Developers using Claude Code/Codex/OpenCode; teams reviewing AI-generated patches; non-technical operators who need to understand what an assistant changed in a site or automation.

Workflow:
1. Initialize agent-aware tracking before a coding session.
2. Let the agent work normally while each step is recorded.
3. Review final changes grouped by prompt/agent step.
4. Use provenance blame to identify the instruction that introduced a bad line.
5. Keep the session log with the PR so reviewers can inspect intent, not just output.

Prompt/tool:
"After this agent session, summarize every changed file with the prompt or step that caused it. Highlight lines whose provenance does not match the approved plan."

Risk / when not to use:
Provenance can become noise if every tiny model action is logged without useful grouping. It also does not prove correctness; it only improves traceability.

## 5. Mobile-app agents need a "permission sandwich" before touching real accounts

Idea:
Mobile-control agents are becoming easy enough to point at normal consumer apps. The useful workflow is not "let AI order tacos"; it is a guarded pattern for app automation where the agent can navigate, but risky account/payment/identity steps require evidence and human confirmation.

Source:
App Use — AI agents controlling iOS/Android apps through Appium-style automation — https://github.com/erickjtorres/app-use

Why unusual:
Most computer-use discussion focuses on browsers and desktops. Mobile apps contain the messy normal-person workflows: delivery, banking, travel, health, messages, subscriptions, and locked-down UIs.

Who it helps:
Developers building mobile automation; normal people experimenting with assistants for errands; small teams testing their own mobile app flows.

Workflow:
1. Allow read-only navigation and information gathering freely.
2. Before a risky tap, require an evidence table: app identity, account, recipient/vendor, amount, item, irreversible effect.
3. Ask the human to confirm the final action in plain language.
4. After completion, capture a receipt/screenshot/log and stop the session.
5. Never store credentials or let the model invent missing UI state.

Prompt/tool:
"You may navigate the app and prepare the action, but before tapping any payment, send, delete, subscribe, or account-change control, show me the exact evidence and ask for confirmation."

Risk / when not to use:
High-risk domains like banking, medical, legal, or private messages need much stronger controls. Screen-reading agents can misread state; mobile UI automation is brittle.
