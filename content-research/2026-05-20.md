## Raw ideas

### 1. Browser-level TDD loop for coding agents

Idea: Turn feature requests into executable browser acceptance tests before asking an agent to build the app.
Source: arXiv cs.SE — “From Runnable to Shippable: Multi-Agent Test-Driven Development for Generating Full-Stack Web Applications from Requirements” — https://arxiv.org/abs/2605.17242
Why unusual: Most coding-agent workflows stop at “the code compiles.” This one treats web correctness as something that must be deployed, clicked through, observed, and repaired with structured failure reports.
Who it helps: Developers shipping AI-generated internal tools, MVPs, admin panels, forms, onboarding flows, and small SaaS features.
Workflow: Requirement → browser-level acceptance tests → generated app → deployed runtime → simulated user journey → observed UI failure → repair instruction → rerun.
Prompt/tool: “Before writing code, convert this requirement into 5 browser-level acceptance tests. After implementation, run the deployed app against those tests and translate each observed UI failure into one concrete repair instruction.”
Risk / when not to use: Too heavy for throwaway prototypes; token/runtime cost can balloon if the app is vague or the test protocol fights the model.

### 2. Recovery ladder for brittle tool-using agents

Idea: Put a small harness around an agent so failures escalate through deterministic recovery before asking the model to improvise again.
Source: arXiv cs.AI — “CAX-Agent: A Lightweight Agent Harness for Reliable APDL Automation” — https://arxiv.org/abs/2605.15218
Why unusual: The interesting bit is not MAPDL itself; it is the middleware pattern: tool lifecycle, workflow state, failure detection, and an explicit recovery ladder.
Who it helps: Developers wrapping brittle CLIs, CAD/CAE tools, browser automation, spreadsheets, ERP tools, or local scripts with AI.
Workflow: User task → agent harness → domain tool → failure classification → deterministic patch/rule fix → model regeneration → context enrichment → human escalation.
Prompt/tool: “Design a recovery ladder for this agent: deterministic fixes first, model retry second, context enrichment third, human escalation last. For each rung, define the failure signals and stop conditions.”
Risk / when not to use: Domain-specific paper; do not generalize the performance claims without testing. Harnesses add overhead if the tool path is already simple and safe.

### 3. Diagnose failed GUI-agent runs instead of blindly retrying

Idea: When a GUI evaluation fails, reuse the trajectory and run targeted probes to distinguish evaluator error from a real product bug.
Source: arXiv cs.SE — “DiagEval: Trajectory-Conditioned Diagnosis for Reliable Software Evaluation with GUI Agents” — https://arxiv.org/abs/2605.17439
Why unusual: It reframes UI-agent failure as diagnosis over a state trajectory, not a binary pass/fail screenshot.
Who it helps: Developers evaluating AI-built UIs, QA teams using browser agents, and small teams that waste time rerunning flaky agent tests.
Workflow: Failed rollout → inspect trajectory → identify ambiguous step → run targeted probes → classify as evaluator mistake, unreachable state, or real defect → aggregate diagnostic signal.
Prompt/tool: “Given this failed UI trajectory, do not retry from scratch. Design three probes: one for evaluator error, one for unreachable UI state, and one for actual app defect. Report which hypothesis each probe supports.”
Risk / when not to use: Works after a failure; it does not replace good acceptance-test design or broader state coverage.
Score: 8/10

### 4. Memory stack for security repair agents

Idea: Security-fixing agents should remember prior fixes, repo conventions, and failed validation paths instead of treating every vulnerability as a fresh one-shot patch.
Source: arXiv cs.SE — “MemRepair: Hierarchical Memory for Agentic Repository-Level Vulnerability Repair” — https://arxiv.org/abs/2605.17444
Why unusual: The memory object is practical: prior patches, reusable security patterns, and failure-to-success refinement trajectories.
Who it helps: Security engineers, maintainers, and teams experimenting with automated vulnerability repair across multi-file repos.
Workflow: Vulnerability report → retrieve similar prior fixes and repo security conventions → patch → validate → store failed refinement path → revise with runtime evidence → human security review.
Prompt/tool: “Before patching, retrieve similar prior fixes, the repo’s security conventions, and previous failed validation attempts. After validation, write down what changed from failure to success so the next repair starts smarter.”
Risk / when not to use: Security patches still need human review. Bad memory can institutionalize bad historical patterns if not curated.
Score: 8/10

### 5. SaaS-agent preflight for long-horizon engineering tasks

Idea: Before letting a coding agent touch a realistic SaaS task, force it through a cross-service preflight: auth, data model, integrations, tests, deployment risks.
Source: arXiv cs.SE — “SaaSBench: Exploring the Boundaries of Coding Agents in Long-Horizon Enterprise SaaS Engineering” — https://arxiv.org/abs/2605.17526
Why unusual: The paper’s value is less “new benchmark” and more a reminder that real SaaS work is heterogeneous, stateful, and integration-heavy — exactly where agents overclaim.
Who it helps: Small teams deciding whether a coding agent should handle a feature, a slice of a feature, or only generate a plan.
Workflow: Task brief → cross-service dependency map → data/auth impact check → test surface list → deployment/rollback risks → only then code generation.
Prompt/tool: “Assess this as enterprise SaaS work before editing: list cross-service dependencies, data model impacts, auth/permission risks, integration surfaces, tests required, and deployment rollback plan.”
Risk / when not to use: Benchmark-oriented rather than a direct tool. Better as an evaluation/preflight checklist than a standalone post unless paired with concrete examples.
Score: 7/10

## Candidate angle

Title: Stop asking coding agents to “build it” — make them prove the user journey
Short answer: The practical leap is not a smarter code generator; it is a loop that turns requirements into browser-level tests, deploys the result, observes failures, and feeds structured repair instructions back to the agent.
Flow: Requirement → Acceptance tests → Build → Deploy → Browser simulation → Failure report → Repair loop → Human approval
Visual component idea: `FlowGraphic` for the loop plus `DoDont`: don’t trust compile success; do test the actual user journey.
