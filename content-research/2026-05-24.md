## Ideas

Idea: Visual judge loops for physical or taste-constrained decisions
Source: arXiv — “TO-Agents: A Multi-Agent AI Pipeline for Preference-Guided Topology Optimization” https://arxiv.org/abs/2605.21622
Why unusual: It does not ask AI to magically design the object. It translates fuzzy preference into solver parameters, renders multi-view candidates, uses a separate visual judge, then revises only the parameters tied to the critique.
Who it helps: CAD/CAE teams, makers, product designers, 3D-print hobbyists, home/interior planners, and normal people choosing between visual/physical options with constraints.
Workflow: Write taste + hard constraints → generate or render 3 candidates → critique each from multiple views → score against constraints → revise generation/solver parameters, not just the prompt → keep the best candidate plus a “why this might fail” note.
Prompt/tool: “Judge these options against: [taste], [hard constraints], [cost/safety/use constraints]. Return: score, violated constraints, one generator/solver parameter to change, and one reason not to change it.”
Risk / when not to use: Visual judges can reward style and miss physics, safety, cost, accessibility, or manufacturing limits; keep human review and hard constraints outside the model.

Idea: Give agents read-only database access through a policy layer, not raw credentials
Source: HN / GitHub — SafeDB MCP, read-only SQL MCP with schema/table allowlists, query validation, row caps, masking, and audit logs https://github.com/narekmalk/safedb-mcp
Why unusual: The workflow is not “connect the agent to prod.” It inserts a narrow database lens where the agent can inspect facts, explain queries, and answer product questions without mutation rights.
Who it helps: Developers, indie SaaS operators, support teams, product marketers, and normal site owners who want AI to answer “what changed?” from data without handing it write access.
Workflow: Create a read-only DB user → expose only safe schemas/tables → cap rows and mask sensitive columns → give the agent tiny tools like `describe_table`, `explain_query`, and `run_readonly_query` → require a human before any product/email/pricing action based on the result.
Prompt/tool: “Before changing this page or support answer, inspect only the allowed analytics/support tables. Return the query, capped result summary, uncertainty, and the smallest safe recommendation. Do not mutate data.”
Risk / when not to use: Read-only is not automatically private; sensitive rows can still leak. Avoid raw PII, production secrets, low-volume overfitting, and any workflow where the agent can act externally without approval.

Idea: Put a command-risk firewall between coding agents and the terminal
Source: HN / GitHub — Terminal Guardian MCP, command risk analysis and sandboxed terminal access for AI assistants https://github.com/7Majesty-M/terminal-guardian-mcp
Why unusual: It turns terminal access into a graded permission system: safe commands run, warning commands require confirmation, dangerous commands are blocked by default, and environment secrets are masked.
Who it helps: Developers using Claude/Codex/OpenCode, small teams giving agents repo access, and non-devs asking assistants to maintain websites or local automations.
Workflow: Route terminal commands through a guardian → classify safe/warning/dangerous/blocked → require confirmation for deletes/stops/kills → mask secrets in env output → log every blocked or confirmed command → turn repeated false positives into explicit project rules.
Prompt/tool: “You may run safe inspection commands. Before any delete, install, process kill, service restart, network pipe, or permission change, explain the exact command, risk level, rollback path, and wait for confirmation.”
Risk / when not to use: Static risk labels can be gamed or miss context; this is a guardrail, not proof. Still use backups, dry-runs, least privilege, and human approval for destructive work.

Idea: Test whether docs are agent-readable by making simulated developers use them
Source: HN / GitHub — dari-docs, simulated developer agents that attempt tasks using only your docs and report where they get stuck https://github.com/mupt-ai/dari-docs
Why unusual: It treats documentation quality as an executable test: if agents cannot complete realistic tasks from the docs, the docs are missing setup, terminology, or decision context.
Who it helps: Developer-tool teams, open-source maintainers, internal platform teams, and normal people writing SOPs for assistants or contractors.
Workflow: Pick 3 real tasks → give an agent only the docs and a clean workspace → capture stuck points, missing assumptions, and wrong commands → patch docs → rerun until the task passes without hidden context.
Prompt/tool: “Use only these docs to complete [task]. When blocked, quote the missing or ambiguous instruction, name the assumption you were forced to make, and propose the smallest docs edit.”
Risk / when not to use: Agents may pass despite docs being poor by guessing from prior knowledge. Run with strict source limits and treat results as friction detection, not universal user proof.

## Pattern watch

- Strongest pattern: agent safety is becoming a set of small, composable policy lenses — read-only data, command risk gates, evidence-capped docs tests — instead of one giant “be careful” prompt.
- Best fresh field-note candidate: SafeDB MCP + Terminal Guardian as “least-privilege tools for useful agents.”
- Normal-people bridge: the same pattern works for databases, websites, home decisions, and SOPs: let the assistant inspect and prepare, but constrain mutation/purchase/send/delete behind evidence and human confirmation.
