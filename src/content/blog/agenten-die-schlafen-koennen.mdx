## Die kurze Version

Ein echter Agent arbeitet nicht die ganze Zeit.

Er wartet.

Auf eine Unterschrift. Auf eine Antwort. Auf Hardware. Auf eine Freigabe. Auf ein Event aus einem anderen System.

Deshalb ist die spannendste Agenten-Architektur nicht “mehr Tools im Prompt”, sondern: **Kann der Agent schlafen, ohne seinen Zustand zu verlieren?**

## Der Denkfehler: Agent = langer Chat

Viele Agenten-Demos funktionieren wie ein sehr langer Chat mit Tools. Das reicht für “lies dieses Repo und schreib einen Fix”.

Es reicht nicht für Prozesse, die Tage dauern:

- HR-Onboarding: warten auf unterschriebene Dokumente
- Einkauf: warten auf Freigabe und Lieferbestätigung
- Rechnungsstreit: warten auf Antwort des Vendors
- Vertrieb: Follow-up erst nach drei Tagen
- Compliance: sammeln, prüfen, eskalieren, wieder warten

Wenn du das als Chat baust, musst du alte Historie wiederholen, pollst Systeme leer oder hältst Jobs künstlich offen. Das ist kein Agent. Das ist ein sehr teurer Timer.

## Mini-Workflow zum Klauen

```txt
Baue diesen Agenten nicht als langen Chat.

Modelliere ihn als State Machine:
1. dauerhafte Zustände
2. erlaubte Übergänge
3. Wake-up-Events
4. Approval Gates
5. State Deltas beim Aufwachen
6. Sub-Agenten für Spezialschritte
7. Tests, die 48h Wartezeit in Sekunden simulieren

Gib außerdem an, wann der Agent NICHTS tun darf.
```

## Wann lohnt sich das?

Nicht jeder Agent braucht Schlaf. Ein Code-Fix, eine Zusammenfassung, ein kurzer Research-Task: alles okay als Session.

Aber sobald der Workflow auf Menschen, E-Mails, Lieferungen, Freigaben oder externe Systeme wartet, brauchst du ein anderes Modell.

## Der bessere Agent wirkt langweilig

Das ist der Punkt.

Ein guter Langläufer ist nicht dauernd “aktiv”. Er speichert Zustand, geht schlafen und wacht nur auf, wenn wirklich etwas passiert ist.

Vielleicht ist das die wichtigste Lektion für Agenten 2026: **Autonomie sieht oft aus wie diszipliniertes Nichtstun.**
