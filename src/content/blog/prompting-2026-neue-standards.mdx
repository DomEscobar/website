## Kurz erklärt

Prompting ist nicht tot. Es wurde nur aus der Chatbox befördert.

Früher war ein Prompt oft der ganze Workflow: ein langer Text, ein bisschen Rolle, ein paar Beispiele, dann hoffen.

2026 ist ein guter Prompt eher die **Startfläche eines Systems**: Er sagt dem Modell nicht nur, was es tun soll, sondern auch, welchen Kontext es benutzen darf, welche Tools zählen, welches Format rauskommen muss, wie Unsicherheit aussieht und woran Erfolg gemessen wird.

## Warum der alte Prompting-Vibe bricht

Der alte Vibe war:

```txt
Act as a world-class expert.
Think step by step.
Be accurate.
Return JSON.
```

Das war nicht komplett falsch. Aber es ist zu grob.

Heute ist das Problem nicht mehr nur: “Wie schreibe ich die perfekte Anweisung?”

Das Problem ist:

```txt
Welche Informationen darf das Modell sehen?
Welche Informationen sind nur Daten und keine Anweisungen?
Welche Tools darf es benutzen?
Welches Format muss maschinell validierbar sein?
Wie merke ich, ob der Output schlechter geworden ist?
Wann muss der Agent stoppen?
```

Das ist kein Copywriting-Problem mehr. Das ist Systemdesign.

## Die neue Formel

```txt
klarer Task Contract
+ kuratiertes Kontext-Paket
+ Output-Schema
+ Tool-Regeln
+ Reasoning-Policy
+ Eval / Akzeptanzcheck
= moderner Prompt
```

Ein moderner Prompt beantwortet nicht nur “Was soll passieren?”, sondern auch:

- Was ist das Ziel?
- Was ist **nicht** Teil des Ziels?
- Welche Fakten sind verbindlich?
- Welche Eingaben sind untrusted?
- Welche Tools zählen als Evidenz?
- Wie sieht ein Blocker aus?
- Wann darf Erfolg behauptet werden?

## Der Task Contract

Wenn du nur eine Struktur übernimmst, nimm diese:

```xml
<role>
Du bist ein präziser Assistent für [Domain].
</role>

<task>
Erstelle [Output] für [Zielgruppe], damit [Ziel].
</task>

<context>
Nur die relevanten Fakten, Quellen, Entscheidungen und Constraints.
</context>

<constraints>
- Nutze keine unbelegten Behauptungen.
- Trenne Fakten von Empfehlungen.
- Wenn Informationen fehlen, gib BLOCKED zurück.
</constraints>

<tools>
- Nutze Tools für aktuelle oder veränderliche Fakten.
- Simuliere keine Tool-Ergebnisse.
- Verifiziere vor finaler Erfolgsmeldung.
</tools>

<output_format>
1. Kurzantwort
2. Evidenz
3. Empfehlung
4. Caveats oder Blocker
</output_format>
```

Das wirkt weniger sexy als ein viraler Prompt. Aber es ist viel robuster.

## Reasoning-Modelle brauchen andere Prompts

Der Satz “Think step by step” war lange der Standard-Tipp. Für viele alte Modelle konnte das helfen.

Bei modernen Reasoning-Modellen ist die bessere Regel:

<KeyPoints
  title="Reasoning sauber steuern"
  items={[
    'Bitte nicht blind vollständige Chain-of-Thought erzwingen.',
    'Nutze Reasoning-Effort oder Thinking-Budget, wenn das Modell es anbietet.',
    'Fordere lieber eine kurze Begründung, Annahmenliste oder Verifikationsnotiz.',
    'Für schwierige Entscheidungen: mehrere Optionen + Kriterien + Empfehlung statt Gedankenprotokoll.',
  ]}
/>

Der Punkt ist nicht, dass Modelle nicht denken sollen. Der Punkt ist: Du willst **nutzbare Kontrolle**, nicht ein langes Gedanken-Theater.

## Structured Outputs schlagen “mach JSON”

Wenn Menschen den Output lesen, ist Markdown okay.

Wenn Software den Output verarbeitet, brauchst du ein Schema.

```txt
Schwach:
Bitte gib valides JSON zurück.

Stärker:
Gib ein Objekt zurück, das exakt diesem JSON Schema entspricht.
Wenn ein Feld unbekannt ist, setze unknown: true und erkläre warum.
```

Warum?

Weil “valides JSON” nur sagt: Der Text ist parsebar.

Ein Schema sagt: Der Output hat die richtige Form.

Das ist ein riesiger Unterschied, wenn der nächste Schritt ein Import, ein API-Call, ein Agent-Plan oder ein Eval ist.

## Tool-Regeln sind Teil des Prompts

Sobald ein Modell Tools benutzen kann, reicht eine Aufgabenbeschreibung nicht mehr.

Du brauchst Regeln wie:

```txt
- Lies Dateien, bevor du sie bearbeitest.
- Prüfe aktuelle Fakten mit Tools, statt sie aus Erinnerung zu beantworten.
- Frage vor externen, destruktiven oder schwer rückgängig zu machenden Aktionen.
- Behaupte Erfolg erst nach Build/Test/Live-Check.
- Wenn ein Tool fehlschlägt, markiere den Schritt als Blocker statt zu raten.
```

Das klingt langweilig. Genau deshalb funktioniert es.

## Lange Kontexte sind nicht automatisch gute Kontexte

Long Context ist mächtig. Aber “mehr Kontext” ist nicht dasselbe wie “besserer Kontext”.

Wenn du dem Modell 800 Dateien gibst, hat es zwar mehr Material — aber auch mehr Rauschen.

Besser ist:

- relevante Dokumente nach oben
- Aufgabe und Frage klar ans Ende
- Quellen sauber trennen
- untrusted Inhalte labeln
- unnötige Historie rauswerfen
- Retrieval/Evals testen

Der neue Skill ist nicht, alles ins Fenster zu kippen. Der neue Skill ist, den Kontext so zu bauen, dass das Modell nicht suchen, raten und halluzinieren muss.

## Prompt Chaining ist kein Selbstzweck

Mehrere Prompts hintereinander sind sinnvoll, wenn du Zwischenstände sehen oder kontrollieren musst.

Zum Beispiel:

```txt
1. Quellen extrahieren
2. Claims gegen Quellen prüfen
3. Entwurf schreiben
4. Review gegen Kriterien
5. Finalisieren
```

Aber Chaining ist nicht automatisch besser. Manchmal reicht ein guter Task Contract plus Schema plus Eval.

Die Frage ist nicht: “Ein Prompt oder fünf?”

Die Frage ist: **Welche Zwischenergebnisse müssen überprüfbar sein?**

## Der wichtigste Teil: Evals

Ein Prompt, der Geld, Code, Support, SEO oder Produktentscheidungen beeinflusst, braucht Tests.

```txt
Golden Cases
+ Edge Cases
+ Injection Cases
+ Schema Checks
+ Fakten-/Quellen-Checks
+ Regression nach Prompt-Änderungen
```

Ohne Evals ist Prompting Bauchgefühl.

Mit Evals wird Prompting Produktlogik.

## Fazit

Prompting ist nicht verschwunden.

Es ist erwachsen geworden.

Der gute Prompt 2026 ist kein einzelner genialer Satz. Er ist die dünne, klare Schnittstelle zwischen Mensch, Modell, Kontext, Tools, State und Verifikation.

Oder kürzer:

```txt
Prompting ist nicht mehr der ganze Workflow.
Prompting ist das Interface zum Workflow.
```
