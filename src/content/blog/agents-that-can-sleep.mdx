## Short answer

A real agent is not working all the time.

It waits.

For a signature. For a reply. For hardware. For approval. For an event from another system.

So the interesting agent architecture is not “more tools in the prompt.” It is: **can the agent sleep without losing its place?**

## The mistake: agent = long chat

Many agent demos work like a very long chat with tools. That is fine for “read this repo and fix the bug.”

It is not enough for workflows that take days:

- HR onboarding: wait for signed documents
- procurement: wait for approval and delivery confirmation
- invoice disputes: wait for the vendor reply
- sales: follow up after three days
- compliance: collect, review, escalate, wait again

If you model that as chat, you either replay old history, poll systems constantly, or keep jobs open for no reason. That is not an agent. That is an expensive timer.

## Tiny workflow to steal

```txt
Do not build this agent as one long chat.

Model it as a state machine:
1. durable states
2. allowed transitions
3. wake-up events
4. approval gates
5. state deltas on resume
6. sub-agents for specialized steps
7. tests that simulate 48h of idle time in seconds

Also specify when the agent must do nothing.
```

## When it is worth it

Not every agent needs sleep. A code fix, a summary, a short research task: a session is fine.

But once the workflow waits on humans, email, deliveries, approvals, or external systems, you need a different model.

## The better agent looks boring

That is the point.

A good long-running agent is not constantly “active.” It stores state, goes dormant, and wakes only when something actually happens.

Maybe that is the most useful agent lesson for 2026: **autonomy often looks like disciplined inaction.**
