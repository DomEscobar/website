## Short answer

The worst prompt for a frontend agent is: **“make it look better.”**

Not because AI cannot build UI. It can write Tailwind, refactor components, clean SVGs, and wire up pages. The problem is that “looks better” is not a test. If the QA is vibes, the output will be vibes.

The better question is:

> Which parts of this design can be measured?

## Translate “it looks wrong”

A human says: “The spacing feels off.”

A useful pipeline says:

- expected gap: `32px`
- actual gap: `24px`
- expected font size: `18px`
- actual font size: `16px`
- this animation is unsafe in Safari

That is the difference between opinion and feedback an agent can act on.

## Tiny workflow to steal

```txt
Implement this UI section.
Do not rely on visual similarity alone.

Return:
1. extracted design values: spacing, type, dimensions, colors, radius
2. component plan
3. browser computed-style checks after implementation
4. exact mismatches vs the source design
5. what still needs human taste review
```

## When it is worth it

You do not need this for every button. It is worth it when the UI work repeats: landing page sections, design system migrations, Figma-to-code pipelines, client websites, or components that must stay consistent.

## Humans still matter

Measurable QA does not replace a good designer. It removes the annoying part of review: “Is the gap correct? Is the type size right? Does Safari break?”

Then the human can do the human thing: decide whether the UI is actually good.
