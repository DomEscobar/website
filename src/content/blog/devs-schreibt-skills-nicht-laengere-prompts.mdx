## Die kurze Version

Wenn dein Coding Agent Mist baut, ist die erste Reaktion oft:

> „Dann schreibe ich halt einen besseren Prompt.“

Und dann wächst der Prompt. Mehr Regeln. Mehr Ausnahmen. Mehr „Bitte wirklich beachten“. Mehr Bulletpoints, die irgendwo in der Mitte verschwinden.

Das Problem: Ein Sprachmodell liest nicht wie ein Mensch mit Gewissen. Es arbeitet über Tokens, Attention und ein begrenztes Context Window. Ein langer Prompt ist nicht automatisch eine bessere Steuerung. Oft ist er nur ein größerer Raum, in dem die wichtigste Regel verloren geht.

Für Entwickler ist die bessere Abstraktion: **Skills statt Mega-Prompts.**

## Warum das technisch Sinn macht

Transformer-Modelle routen Information über Attention: Jeder Token entscheidet, welche anderen Tokens gerade wichtig sind. Volle Attention ist mächtig, aber teuer — und bei langen Kontexten nicht perfekt.

Die Forschung zeigt zwei Dinge, die jeder Entwickler beim Prompten kennen sollte:

1. **Context Window ist Input-Budget, kein echtes Gedächtnis.** Nur weil ein Modell 100k Tokens annehmen kann, heißt das nicht, dass es jeden Satz zuverlässig benutzt.
2. **Wichtige Infos können im langen Kontext untergehen.** In „Lost in the Middle“ schneiden Modelle oft besser ab, wenn relevante Informationen am Anfang oder Ende stehen — und schlechter, wenn sie in der Mitte begraben sind.

Das heißt praktisch:

```txt
mehr Prompt ≠ mehr Kontrolle
mehr Kontext ≠ mehr Verständnis
mehr Regeln ≠ mehr Verlässlichkeit
```

Ein Skill löst das nicht magisch. Aber er macht die Steuerung kleiner, lokaler und testbarer.

## Was ist ein Skill?

Ein Skill ist kein magischer Prompt. Ein Skill ist eher wie eine kleine interne Developer-Doku, die ein Agent wirklich benutzen kann.

```txt
Skill =
  wann benutzen?
+ was ist der Input?
+ welche Schritte sind Pflicht?
+ welche Tools/Dateien sind erlaubt?
+ wann muss der Agent stoppen oder fragen?
+ wie wird das Ergebnis verifiziert?
```

Für Menschen klingt das banal. Für Agenten ist es der Unterschied zwischen „mach mal Code Review“ und einem wiederholbaren Arbeitsablauf.

## Beispiel: aus Prompt wird Skill

Schlechter Prompt:

```txt
Review bitte den Code gründlich, achte auf Bugs, Sicherheit,
Performance, Style, Tests und sag mir alles Wichtige.
```

Besserer Skill:

```md
# Code Review Skill

Use when: user asks for review, PR feedback, risky diff, or regression check.

Inputs:
- changed files
- test output if available
- project rules / AGENTS.md

Process:
1. Read the diff before judging.
2. Identify correctness bugs first.
3. Check security/privacy risks.
4. Check missing tests or broken contracts.
5. Only then mention style or cleanup.

Rules:
- Do not request broad rewrites unless necessary.
- Do not flag speculative issues as facts.
- Cite file + line for every finding.

Output:
- Critical findings first
- Then medium/low findings
- Then “no issue found” only if verified

Verification:
- Mention which files were inspected.
- Mention which tests/builds were run or why not.
```

Der zweite Text ist nicht nur länger. Er hat eine andere Form: Trigger, Inputs, Prozess, Regeln, Output, Verification. Das ist näher an Software als an Copywriting.

## Warum gerade Entwickler Skills schreiben sollten

Entwickler verstehen schon, wie man Systeme robust macht:

- kleine Module statt God Object
- Tests statt Hoffnung
- Interfaces statt impliziter Absprachen
- Linting statt „bitte sauber schreiben“
- CI statt manuellem Bauchgefühl

Skills sind genau dieser Shift für Agenten.

## Wann du einen Skill schreiben solltest

Schreib keinen Skill für jede Kleinigkeit. Schreib einen Skill, wenn ein Workflow wiederkommt oder teuer schiefgehen kann.

Gute Kandidaten:

- Code Review
- Debugging
- Release / Deployment
- Security Review
- Datenmigration
- Research mit Quellenpflicht
- UI QA / Screenshot Review
- Writing Style für ein Produkt
- Support-Antworten mit Policies
- Refactoring mit Grenzen

## Die Skill-Template, die du klauen kannst

```md
# <Skill Name>

## Use when
- <Trigger 1>
- <Trigger 2>

## Do not use when
- <Out-of-scope case>
- <Risky case that needs human approval>

## Inputs
- <Required input>
- <Optional input>

## Workflow
1. <First step>
2. <Second step>
3. <Third step>

## Rules
- <Hard constraint>
- <Tool/file permission>
- <Escalation rule>

## Output format
Return:
1. <section>
2. <section>
3. <section>

## Verification
Before final answer:
- <test/check/source>
- <evidence to report>

## Examples
Good:
<short good output>

Bad:
<short bad output and why>
```

## Der eigentliche Mindset-Shift

Prompt Engineering war Phase 1: Menschen lernen, wie sie mit Modellen sprechen.

Context Engineering ist Phase 2: Teams lernen, welche Informationen, Tools und Regeln ins Modellfenster gehören.

Skill Engineering ist Phase 3: Entwickler bauen kleine Arbeitsmodule, damit Agenten nicht jedes Mal neu raten müssen, wie ein Workflow funktioniert.

```txt
Prompt: Bitte mach X.
Context: Hier ist alles, was du für X brauchst.
Skill: So wird X in diesem Team korrekt erledigt.
```

Das ist der Punkt: Skills sind nicht nur bessere Prompts. Skills sind **Betriebswissen in ausführbarer Form**.

## Takeaway

Wenn du heute mit Coding Agents arbeitest, frag nicht nur:

> „Wie schreibe ich bessere Prompts?“

Frag:

> „Welche wiederholbaren Workflows meines Teams verdienen einen Skill?“

Denn der Agent braucht nicht noch einen Roman im Kontextfenster.

Er braucht eine kleine, klare Arbeitsweise, die er zur richtigen Zeit laden kann.
