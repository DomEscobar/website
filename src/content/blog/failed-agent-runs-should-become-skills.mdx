A coding agent fails. You paste the error back in. It fails differently. You paste again. Eventually it passes, or you get tired and fix the last ten percent yourself.

That feels normal now.

It is also wasteful.

The interesting move in [Trace2Skill](https://arxiv.org/abs/2605.21810) is not “try more samples.” It is: when repeated agent runs create traces, the traces can become a better instruction policy. The paper applies this to long-context hardware design tasks, but the workflow is useful for normal software teams too.

If an agent keeps making the same mistake, do not only retry.

Turn the failure into a local skill.

## the short version

A failed run is not trash. It is training data for your next repo instruction.

  title="Trace to skill loop"

The loop is simple:

1. Run the agent on a task with a real verifier: tests, build, lint, browser check, hidden review, or acceptance checklist.
2. Save the useful trace: prompt, changed files, errors, recovery steps, and final proof.
3. Ask a separate pass to extract repeated failure modes and successful habits.
4. Write a tiny `SKILL.md`, `AGENTS.md`, or `CLAUDE.md` rule.
5. Rerun the next task with that rule in context.

This is how a team’s agent setup stops being generic.

## what to mine from the trace

Most logs are noise. The useful parts are the places where the verifier taught you something the prompt did not.

Look for four things:

The skill should not say “be careful.”

It should say something like:

```txt
Before editing generated blog content, inspect the content schema and one recent published post.
After editing, run npm run build.
If the post is draft:true, verify it does not appear in sitemap, RSS, or llms outputs.
Never claim publish-ready if hreflang points to a missing translation.
```

That is small. It is operational. It changes the next run.

## steal this prompt

Use this after a messy agent session.

```txt
You are mining failed agent traces into a local skill.

Inputs:
- original task
- prompts used
- changed files
- command output
- test/build/browser failures
- final successful fix, if any

Produce:
1. recurring failure modes, with evidence from the trace
2. verifier-relevant files and commands
3. recovery moves that actually helped
4. rules to add to SKILL.md / AGENTS.md / CLAUDE.md
5. rules to reject as superstition

Constraints:
- every permanent rule needs evidence
- prefer concrete commands, files, and stop conditions
- keep the final skill under 20 lines
- do not include secrets, private logs, or hidden test details
```

The important line is “rules to reject as superstition.” Failed traces are seductive. A model can turn any coincidence into a ritual if you let it.

This is especially useful for repo-specific work: migrations, generated artifacts, browser flows, release scripts, or anything with hidden assumptions.

The agent needs the scar tissue from the last failure.

## where it fails

Trace-mined skills can go bad in two ways.

First, they can encode superstition. Maybe one retry passed because the dependency cache warmed up.

Second, they can leak things. Logs can contain secrets, customer data, hidden tests, private URLs, or credentials. Sanitize first.
